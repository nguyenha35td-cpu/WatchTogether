## Bug 修复：上传视频无法在另一端播放

### 问题
上传的视频使用 `URL.createObjectURL(file)` 生成 blob URL，该 URL 仅在本地浏览器有效，另一端无法访问导致一直转圈。

### 解决方案
1. 后端添加 multer 文件上传中间件 + Express 静态文件服务（`/uploads` 路由）
2. 前端上传视频文件到 `POST /api/upload`，获取服务器 HTTP URL
3. 所有客户端通过 HTTP URL 访问视频文件，实现跨客户端播放同步
4. 上传时显示"⏳上传中"临时状态，上传完成后替换为正式条目
5. 新增环境变量: `NEXT_PUBLIC_API_URL=http://localhost:3001`

## Bug 修复：暂停时显示转圈 + 复制房间号报错

### 问题1：被同步端暂停时中心显示 loading 转圈
- 原因：`isBuffering` 状态在暂停时未被清除，且中心播放按钮需要 `!isBuffering` 才显示
- 修复：`handleVideoPause` 中清除 `isBuffering`；buffering 转圈仅在 `isPlaying` 时显示；暂停时始终显示播放按钮

### 问题2：房间信息面板首次打开时复制房间号报错
- 原因：`navigator.clipboard.writeText()` 在非 HTTPS 或 Popover 焦点问题下不可用
- 修复：添加 fallback 方案（textarea + execCommand），外层 try/catch 兜底 prompt 手动复制

## 修复：上传视频路径在 Railway 部署后不可用

### 问题
前端代码中 API 地址和 WebSocket 地址 fallback 为 `http://localhost:3001` / `ws://localhost:3001`，部署到 Railway 后无法访问。

### 修复（server/index.js 本身已无问题，实际修改前端）
- `app/page.tsx`：上传请求直接发到 Railway 后端（`https://watchtogether-production-b75c.up.railway.app/api/upload`），绕过 Vercel 4.5MB 请求体限制；视频 URL 也使用完整后端地址
- `hooks/use-websocket.ts`：`NEXT_PUBLIC_WS_URL` fallback 改为自动检测 `window.location`，根据页面协议(http/https)自动选择 ws/wss，使用当前域名和端口
- `next.config.mjs`：添加 rewrites 反向代理，将 `/api/*` 和 `/uploads/*` 请求转发到 Railway 后端（小请求如健康检查等仍可走代理）
